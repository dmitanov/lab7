Цель работы

Изучить поведенческие паттерны проектирования GoF на примере создания консольного приложения, имитирующего систему мониторинга и оповещения о событиях. В ходе выполнения работы необходимо освоить применение паттерна Наблюдатель для организации подписки на события, паттерна Стратегия для выбора формата уведомлений и паттерна Шаблонный метод для задания общего алгоритма обработки событий.


Задание

Требовалось разработать консольную программу, которая моделирует работу системы мониторинга и оповещения о событиях. Приложение должно обеспечивать отслеживание различных метрик системы, таких как загрузка процессора, использование памяти и сетевая активность, а также формирование уведомлений при превышении заданных пороговых значений. Система должна поддерживать разные каналы доставки уведомлений, например вывод в консоль, запись в файл и имитацию отправки по email. Кроме того, необходимо было реализовать разные форматы сообщений: текстовый, JSON и HTML. Архитектура приложения должна обеспечивать возможность добавления новых типов подписчиков, новых форматов сообщений и новых типов событий без существенной переработки уже существующего кода


Теоретическое обоснование

Поведенческие паттерны проектирования используются для организации взаимодействия между объектами и распределения ответственности внутри программы. Их применение позволяет сделать систему более гибкой, расширяемой и удобной для сопровождения. В данной лабораторной работе были использованы паттерны Наблюдатель, Стратегия и Шаблонный метод, которые работают совместно в рамках одной программной системы.

Паттерн Наблюдатель определяет отношение «один ко многим», при котором один объект уведомляет все зависимые объекты об изменении своего состояния. Этот паттерн удобен в системах мониторинга, где при возникновении события необходимо сразу оповестить несколько обработчиков. В рамках данной работы Наблюдатель используется для того, чтобы объект мониторинга сообщал всем подписанным обработчикам о превышении критического значения метрики. В реализации на языке C# для этого применяются делегаты и события, что позволяет удобно организовать подписку и уведомление обработчиков.

Паттерн Стратегия позволяет определить семейство алгоритмов, инкапсулировать каждый из них в отдельный класс и делать их взаимозаменяемыми. В данной работе этот подход применяется к форматированию сообщений. Один и тот же обработчик может использовать разные стратегии представления данных: обычный текст, JSON или HTML. Благодаря этому формат сообщения можно изменять без изменения логики самого обработчика. Такой подход уменьшает связанность кода и делает систему более гибкой.

Паттерн Шаблонный метод определяет общий скелет алгоритма в базовом классе, оставляя реализацию отдельных шагов подклассам. В разрабатываемой системе этот паттерн используется в базовом классе обработчиков событий. Общий порядок работы фиксирован: сначала сообщение формируется, затем отправляется, после чего логируется результат. Конкретные обработчики, такие как вывод в консоль или запись в файл, переопределяют отдельные шаги, но не меняют саму последовательность обработки. Это позволяет избежать дублирования кода и сохранить единый стиль работы всех обработчиков.

Применение этих трёх паттернов в одной системе позволяет добиться модульности, соблюдения принципов расширяемости и удобства дальнейшего развития проекта.


Описание выполненной работы

В процессе выполнения лабораторной работы было создано консольное приложение, моделирующее систему мониторинга и оповещения о событиях.

Сначала был реализован паттерн Наблюдатель. Для этого были созданы классы, описывающие данные метрики и аргументы события. Также был реализован класс EventMonitor, который проверяет значения метрик и генерирует событие при превышении порога. Обработчики подписываются на это событие и автоматически получают уведомление при его возникновении. Таким образом, мониторинг и обработка уведомлений были разделены между собой, что позволило сделать архитектуру слабосвязанной

Далее был реализован паттерн Стратегия. Для этого создан интерфейс IFormatStrategy, определяющий общий метод форматирования сообщения. Затем были разработаны конкретные стратегии форматирования: текстовая, JSON и HTML. Это позволило организовать гибкий механизм представления уведомлений. Один и тот же обработчик может использовать любую из этих стратегий, а при необходимости формат можно заменить даже во время выполнения программы.

После этого был реализован паттерн Шаблонный метод. В качестве базового класса был создан EventHandlerBase, в котором задан общий алгоритм обработки события. Этот алгоритм включает три шага: форматирование сообщения, отправку сообщения и логирование результата. Затем были созданы конкретные обработчики, такие как ConsoleHandler и FileHandler, которые переопределяют необходимые методы и реализуют конкретный способ доставки уведомления. В моей реализации также был добавлен EmailHandler, который имитирует отправку сообщения по электронной почте. Такой подход позволил сохранить общий алгоритм неизменным и при этом сделать поведение конкретных обработчиков различным.

На заключительном этапе в методе Main была организована демонстрация совместной работы всех паттернов. В программе создаётся объект мониторинга, регистрируются несколько обработчиков с разными стратегиями форматирования, после чего моделируются события превышения порогов для различных метрик. В результате одно событие может быть обработано сразу несколькими подписчиками, а уведомления выводятся в разных форматах и через разные каналы.



Результаты работы

В результате выполнения лабораторной работы было разработано консольное приложение, демонстрирующее использование поведенческих паттернов проектирования при создании системы мониторинга и оповещения о событиях.

Проведённая проверка показала, что применение паттерна Наблюдатель позволяет удобно организовать механизм оповещения нескольких обработчиков о наступлении события. Использование паттерна Стратегия обеспечивает гибкость в выборе формата уведомления и упрощает добавление новых способов представления данных. Паттерн Шаблонный метод позволил выделить общий алгоритм обработки событий и избежать дублирования кода в конкретных обработчиках. Совместное использование этих паттернов сделало архитектуру программы более понятной, расширяемой и удобной для сопровождения.

Таким образом, поставленная цель работы была достигнута. В ходе выполнения лабораторной работы были изучены и практически применены паттерны Наблюдатель, Стратегия и Шаблонный метод, а также показана их эффективная совместная работа в рамках единой программной системы мониторинга и оповещения.
